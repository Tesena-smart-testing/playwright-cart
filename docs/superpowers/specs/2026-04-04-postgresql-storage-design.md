# PostgreSQL Storage Migration — Design Spec

**Date:** 2026-04-04
**Status:** Approved

---

## Context

The server currently stores all data as JSON files on disk under `DATA_DIR/{runId}/`. This works for single-host deployments but has limitations: no transactional integrity, no efficient querying, no relational constraints. The goal is to migrate all structured metadata (runs, test results, errors, annotations, attachment metadata) to PostgreSQL while leaving binary files (screenshots, traces, HTML reports) on disk unchanged.

Scope: `packages/server` only. The reporter and web frontend are unaffected. The public API surface does not change.

---

## Decisions Made

| Topic | Decision |
|-------|----------|
| Binary data | Metadata only to DB; binary files remain on disk |
| ORM / query layer | Drizzle ORM with `drizzle-kit` for migrations |
| Migration approach | Replace `storage.ts` in-place; keep same function signatures |
| DB deployment | `postgres:17-alpine` added to `docker-compose.yml` |

---

## Database Schema

Five tables. The `tests` table uses a surrogate `BIGSERIAL` PK so child tables have a clean single-column FK. A `UNIQUE (run_id, test_id)` constraint preserves the natural key for API lookups.

### Enum types

```sql
CREATE TYPE run_status  AS ENUM ('running', 'passed', 'failed', 'interrupted', 'timedOut');
CREATE TYPE test_status AS ENUM ('passed', 'failed', 'timedOut', 'skipped', 'interrupted');
```

### `runs`

```sql
CREATE TABLE runs (
  run_id        TEXT PRIMARY KEY,
  project       TEXT NOT NULL,
  branch        TEXT,
  commit_sha    TEXT,
  started_at    TIMESTAMPTZ NOT NULL,
  completed_at  TIMESTAMPTZ,
  status        run_status NOT NULL DEFAULT 'running',
  report_url    TEXT
);
CREATE INDEX runs_started_at_idx ON runs (started_at DESC);
```

### `tests`

```sql
CREATE TABLE tests (
  id            BIGSERIAL PRIMARY KEY,
  test_id       TEXT NOT NULL,
  run_id        TEXT NOT NULL REFERENCES runs(run_id) ON DELETE CASCADE,
  title         TEXT NOT NULL,
  title_path    TEXT[] NOT NULL,
  location_file TEXT NOT NULL,
  location_line INTEGER NOT NULL,
  location_col  INTEGER NOT NULL,
  status        test_status NOT NULL,
  duration_ms   INTEGER NOT NULL,
  retry         INTEGER NOT NULL DEFAULT 0
);
CREATE UNIQUE INDEX tests_run_test_uniq ON tests (run_id, test_id);
CREATE INDEX tests_run_id_idx ON tests (run_id);
```

### `test_errors`

```sql
CREATE TABLE test_errors (
  id        BIGSERIAL PRIMARY KEY,
  test_pk   BIGINT NOT NULL REFERENCES tests(id) ON DELETE CASCADE,
  position  INTEGER NOT NULL,
  message   TEXT NOT NULL,
  stack     TEXT
);
```

### `test_annotations`

```sql
CREATE TABLE test_annotations (
  id          BIGSERIAL PRIMARY KEY,
  test_pk     BIGINT NOT NULL REFERENCES tests(id) ON DELETE CASCADE,
  position    INTEGER NOT NULL,
  type        TEXT NOT NULL,
  description TEXT
);
```

### `test_attachments` (metadata only; binary files remain on disk)

```sql
CREATE TABLE test_attachments (
  id           BIGSERIAL PRIMARY KEY,
  test_pk      BIGINT NOT NULL REFERENCES tests(id) ON DELETE CASCADE,
  position     INTEGER NOT NULL,
  name         TEXT NOT NULL,
  content_type TEXT NOT NULL,
  filename     TEXT
);
```

## Implementation Notes

### `writeTestResult` must use a transaction

Inserting a test record involves 4 tables (tests + up to 3 child tables). All inserts must be wrapped in a single transaction so a partial failure cannot leave orphaned rows.

### `getTestResults(runId)` — avoid N+1 queries

Fetch all data for a run's tests with 4 queries, then assemble in JavaScript:
1. `SELECT * FROM tests WHERE run_id = $1`
2. `SELECT * FROM test_errors WHERE test_pk = ANY($testIds)`
3. `SELECT * FROM test_annotations WHERE test_pk = ANY($testIds)`
4. `SELECT * FROM test_attachments WHERE test_pk = ANY($testIds)`

Group child rows by `test_pk`, then map to `TestRecord[]`.

### Timestamps

`RunRecord.startedAt` / `completedAt` are ISO 8601 strings in the existing API. PostgreSQL stores them as `TIMESTAMPTZ`. The storage layer converts between `Date` (Drizzle output) and `string` (API shape) at the boundary.

### Environment variables

| Variable | Purpose |
|----------|---------|
| `DATABASE_URL` | PostgreSQL connection string |
| `DATA_DIR` | Still used for binary file paths (attachments, reports) |

Local dev: set `DATABASE_URL` in `.env` (not committed).

## Docker / Deployment

```yaml
# docker-compose.yml additions
postgres:
  image: postgres:17-alpine
  environment:
    POSTGRES_USER: playwright_cart
    POSTGRES_PASSWORD: playwright_cart
    POSTGRES_DB: playwright_cart
  volumes:
    - db_data:/var/lib/postgresql/data
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U playwright_cart"]
    interval: 5s
    timeout: 3s
    retries: 5
  restart: unless-stopped

server:
  depends_on:
    postgres:
      condition: service_healthy
  environment:
    DATABASE_URL: "postgresql://playwright_cart:playwright_cart@postgres:5432/playwright_cart"
```

Migrations run automatically at server startup via `runMigrations()`. They are idempotent — safe to re-run.
